# 아키텍처 및 설계 원칙 스킬 (Architecture & Design Principles)

**본 스킬은 에이전트가 수행하는 모든 리팩토링(코드 수정, 문서 구조화, 로그 표준화 등) 작업의 근간이 되는 최상위 원칙입니다.** 모든 작업은 물리적 증거(숫자)와 단일 진실 공급원(SSOT)에 기반하여 수행되어야 합니다.

## 1. 단일 진실 공급원 (SSOT) 및 문서화 원칙
*   **비즈니스 로직 SSOT**: 모든 비즈니스 규칙과 기술적 결정의 유일한 기준은 **`docs/CRITICAL_LOGIC.md`**입니다.
*   **작업 맥락 SSOT**: 진행 중인 작업의 상태, 히스토리, 체크리스트는 **`docs/memory.md`**에서만 관리하며, **Append-Only(기존 내용 삭제 금지)** 원칙을 엄수합니다.
*   **데이터 원천 일원화**: 공통 상수(Config), 도메인 타입 정의(Types/Schemas), 전역 상태(Zustand)를 중복 정의하지 않고 한 곳에서 참조하도록 강제합니다.
*   **언어 및 형식**: 모든 설명과 주석은 **한국어(Korean)**를 사용하며, **어떠한 경우에도 이모지 사용을 절대 금지**합니다. 핵심 문장은 반드시 **굵게 표시**합니다.

## 2. DDD 및 3-Layer 계층화 아키텍처
모든 기능적 리팩토링은 비즈니스 단위별 폴더 격리(DDD)를 준수하며, 아래의 **3-Layer 패턴**을 강제합니다.
1.  **Definition (정의)**: 도메인 타입, 인터페이스, 변하지 않는 상수 정의.
2.  **Repository (저장소/접근)**: API 호출, 파일 I/O, DB 트랜잭션 등 외부 데이터 소스와 인터랙션하는 계층. (예: `LogRepository`, `api.ts`)
3.  **Service/Logic (서비스/로직)**: 복합적인 비즈니스 로직 처리, React Hooks, UI 상태 처리. (예: `useStation`, `ErrorMonitorService`)
*   **물리적 격리**: 모든 독립형 확장 기능과 도구는 **`plugins/`** 폴더 내에서 격리된 서비스로 구현합니다.

## 3. 리팩토링 및 코드 복잡도 관리 표준
*   **DRY (Don't Repeat Yourself)**: 동일한 로직이 **2회 이상 반복**될 경우 즉시 공통 유틸리티나 훅으로 추상화합니다.
*   **파일 크기 제한**: **Python 파일은 700~1500줄**, **React 컴포넌트는 500줄** 이내로 유지하며, 상한선 초과 시 즉시 파일 분리 리팩토링을 수행합니다.
*   **코드 무결성**: 코드 생략(`// ...`)은 절대 금지하며, 항상 **즉시 실행 가능한 완전한 코드**를 생성하거나 `replace_file_content`를 통해 원자적으로 수정합니다.
*   **로그 표준화**: 모든 로그는 `logs/` 디렉토리 하위에 구조화하여 저장하며, 백엔드(`loguru`)와 프론트엔드(`Tee-Object`)의 기록 방식을 일원화합니다.

## 4. [물리적 증거 확인] 및 검증 프로토콜
**가설과 추측을 배제하고 오직 물리적 데이터로만 대화합니다.**
*   **숫자 기반 보고**: 작업 결과 보고 시 주관적 표현 대신 **[성공 개수 / 전체 개수]**와 같은 객관적 수치를 사용합니다.
*   **검증 루프**: 수정 후 반드시 테스트나 파일 읽기(`Get-Content`)를 통해 물리적 상태를 확인하고, 성공할 때까지 독립적으로 반복합니다.
*   **터미널 로그 노출**: 모든 검증 단계의 터미널 실행 기록은 사용자에게 투명하게 공개되어야 합니다.

## 5. 핵심 기술 명세 (Critical Logic 준수)
*   **시간대 표준**: 모든 시간 처리는 **KST (Asia/Seoul)** 단일 시간대를 사용합니다.
*   **API 안정성**: 모든 데이터 호출 훅에는 **500ms lastFetchRef 가드**를 적용하여 불필요한 중복 요청을 방지합니다.
*   **DB 무결성**: Supabase/DB 수정 시 **'Update 후 별도 Select'**의 2단계 패턴을 사용하여 실제 반영된 데이터를 확인합니다.
*   **감사 및 보안**: 민감한 정보(환자 성함 등)는 항상 **마스킹** 처리하며, 전실/퇴원 등 중요 동작은 반드시 **`audit_logs`**를 남깁니다.
